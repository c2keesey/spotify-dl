# Crate v3 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development to implement this plan task-by-task.

**Goal:** Close all 22 open beads — backend truth about the library's real state, a full-width track browser, a decoupled BPM graph, file-based set persistence, and the duplicates screen the project was started for.

**Architecture:** The rekordbox master.db stays the analysis engine and is read constantly, written rarely and only additively. A new `setfile.py` module makes an m3u8-plus-JSON-sidecar the single artifact that is simultaneously Crate's saved set, its export to other DJ software, and its mapping back to the rekordbox playlist an export produced. The frontend goes full-bleed on DJ Sets.

**Tech Stack:** FastAPI + pyrekordbox + ffmpeg (backend); React 19 + Vite + TS + Tailwind + shadcn/ui + TanStack Query + dnd-kit (frontend); pytest + vitest.

## Global Constraints

These bind every task. Copied verbatim from the standing project rules.

- **Additive only.** "for safety make sure it pulls in to new playlist or collection or whatever always to avoid overwrites". Never modify or delete an existing rekordbox playlist or track row. New playlists only; uniquify the name on collision.
- **Dedup first.** "do a dup check always frist too". Every import path runs dedup before writing.
- **Never write master.db while rekordbox is running.** Raise `RekordboxRunning` → HTTP 409. Back up master.db before every write.
- **Crate never relocates or deletes the user's audio files.** Missing-file detection reports; it does not repair.
- Subagent model policy: opus mostly, sonnet sometimes, fable very selectively. **Never haiku.**
- No file over ~200 lines. `bun run build` and `uv run pytest` must pass as a task gate.
- Design decisions locked by the user: sets reference tracks by **both** rekordbox content id **and** absolute path; Crate **maps** each export to the playlist it produced and can **read existing playlists in read-only**; on-disk format is **m3u8 + JSON sidecar**.

## Known ground truth (measured 2026-07-08)

- 1437 tracks in the collection; **926 have a `FolderPath` that does not exist on disk** (files moved to `old_downloads/`). Some `FolderPath` values are `spotify:track:…` URIs, not paths at all.
- `load_tracks()` opens the DB and walks every row on **every request**. Adding 1437 `stat()` calls per request is not acceptable — file-existence must be cached with an mtime/TTL guard.
- `dj.get_energy()` returns `None` for missing file, ffmpeg failure, and parse failure alike. The API cannot distinguish them.
- `EnergyScope.tsx:32-34` early-returns before the SVG mounts, so the BPM polyline at `:142-149` never renders without ≥2 energy values.

### Task 1: File presence + genre in the track record

**Files:** Modify `spotify_dl/rekordbox.py`, `spotify_dl/web.py`. Test `tests/test_rekordbox.py`.

**Interfaces produced:**
- `rekordbox.file_state(path: str) -> str` — one of `"present"`, `"missing"`, `"unmounted"`, `"not_a_file"`. `"not_a_file"` covers `spotify:track:…` URIs and any non-absolute path. `"unmounted"` when the path is under a `/Volumes/<name>` whose root does not exist.
- `_record(c)` gains `"genre": str | None` (from `c.Genre.Name` if present, else `None`) and `"file_state": str`.
- `rekordbox.load_tracks()` unchanged signature; internally uses a module-level presence cache keyed by path with a 60s TTL, so a request does not re-`stat()` 1437 files.
- `GET /api/dj/tracks` gains `genre: str = ""` and `file_state: str = ""` filter params.
- `GET /api/dj/status` gains `"missing": int`, `"unmounted": int`, `"not_a_file": int`.

Scope discipline: **genre only.** Do not add rating, color, comment, or My Tag.

### Task 2: Energy reports why it failed

**Files:** Modify `spotify_dl/dj.py`, `spotify_dl/web.py`. Test `tests/test_dj.py`.

**Interfaces produced:**
- `dj.measure_energy(path, cache_file=None) -> dict` returning `{"lufs": float｜None, "state": "measured"|"missing"|"failed"}`.
- `dj.get_energy` stays as a thin wrapper returning `lufs` only, so existing callers and the cache format are untouched.
- `POST /api/dj/energy` returns `{"energy": {id: float|None}, "state": {id: "measured"|"missing"|"failed"}}`. The `energy` map keeps its exact current shape — the frontend contract does not break.

Keep the `threading.Lock` discipline around cache read/write; ffmpeg still runs unlocked.

### Task 3: not_imported honors fuzzy dedup, auto_import scopes to its own job

**Files:** Modify `spotify_dl/web.py`, `spotify_dl/rekordbox.py`. Test `tests/test_web.py`.

- sdl-i1j: `/api/dj/status`'s `not_imported` count currently compares exact paths only, so files that always skip as fuzzy dupes keep the Import button alive forever. Count with the same `find_duplicates` fuzzy logic the importer uses.
- sdl-dig: `run_job`'s auto-import hook currently offers the whole downloads tree. Scope it to the files the job itself produced.
- sdl-jzw: the misc-hardening items already recorded on that bead.

**Preserve:** the autouse `_no_real_auto_import` fixture and the `REAL_AUTO_IMPORT` sentinel test. A test must never be able to bulk-write the real database.

## Wave 3 — Set files and export (sequential; both touch `web.py`)

### Task 7: `setfile.py` — m3u8 + JSON sidecar
**Files:** Create `spotify_dl/setfile.py`, `tests/test_setfile.py`. Modify `spotify_dl/web.py`.
**Beads:** sdl-90w.

**Interfaces produced:**
- `setfile.save(dir, name, tracks) -> Path` writes `<name>.m3u8` (`#EXTM3U`, `#EXTINF:<secs>,<artist> - <title>`, then the absolute path per track, in order) and `<name>.json` (`{"name", "created_at", "tracks": [{"id", "path"}], "rekordbox_playlist_id": null}`).
- `setfile.load(path) -> dict` reads the sidecar, falling back to parsing the m3u8 when the sidecar is absent (so a hand-made m3u8 opens).
- `setfile.list_sets(dir) -> list[dict]`
- `setfile.to_rekordbox_xml(tracks, playlist_name) -> str` — a `DJ_PLAYLISTS` collection + playlist node carrying BPM, key, duration. Importable by rekordbox.
- `POST /api/dj/export/m3u8` and `POST /api/dj/export/xml`. **Neither touches master.db, so neither takes the rekordbox-running guard.**

### Task 8: Duplicate cleanup screen
**Files:** Modify `spotify_dl/rekordbox.py`, `spotify_dl/web.py`. Create `frontend/src/pages/DjSets/Duplicates.tsx`.
**Beads:** sdl-qbj.

- `GET /api/dj/duplicates` groups collection tracks by exact path and by fuzzy artist/title/duration, reusing `find_duplicates`' matching. Each group reports its match reason and the compared field values.
- UI distinguishes exact from fuzzy groups. **Never preselect a copy for removal in a fuzzy group** — fuzzy matches are guesses.
- Output is a review artifact: export the flagged copies as a new uniquified "Duplicates — review" playlist the user deletes by hand inside rekordbox. Crate deletes nothing.

## Wave 4 — Audition (sequential; touches `web.py`)

### Task 9: Audition playback
**Files:** Modify `spotify_dl/web.py`. Create `frontend/src/pages/DjSets/Audition.tsx`.
**Beads:** sdl-awy.

- `GET /api/dj/audio/{track_id}` resolves the id against the rekordbox collection and streams that file with HTTP range support. **The endpoint never accepts a client-supplied path.** Anything not in the collection is a 404. This is the whole security surface — get it right.
- One track plays at a time. Starts at a sensible offset (~⅓ in), not 0:00. Click or spacebar toggles. Missing file → visible error, never a silent no-op.
- No waveform, no scrubbing, no cue points.

## Wave 5 — Set management (depends on Task 7)

### Task 10: Saved sets, mapping, and read-in
**Files:** Modify `spotify_dl/{setfile,rekordbox,web}.py`. Create `frontend/src/pages/DjSets/SetLibrary.tsx`.
**Beads:** sdl-0bn.

- Sets live in a `sets/` directory beside the downloads. List, open into the rail, rename, duplicate, delete (deletes the Crate set file — never the rekordbox playlist).
- `rekordbox_playlist_id` in the sidecar records which playlist an export produced, so the UI shows "exported" and can detect drift. **Re-export creates a new uniquified playlist and updates the mapping — it never mutates the old one.**
- `GET /api/dj/playlists` reads existing rekordbox playlists in as read-only sets the user can fork into a new Crate set. Purely additive, zero write risk.
- Resolution order on open: content id first, absolute path as fallback. Report anything unresolvable rather than silently dropping it.
